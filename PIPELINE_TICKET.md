# Engineering Ticket — Encoded Carousel Automation Pipeline

**Project:** Encoded.Ai — Instagram Carousel System
**Repo:** https://github.com/MAJAJECAR/encoded-carousels
**Reference app:** https://encoded-carousels-main.netlify.app/
**Priority:** High
**Type:** Feature — Full Pipeline Automation

---

## Background

We currently have a working carousel system built as static HTML files that are
screenshotted locally using Playwright. The full system is documented in `CLAUDE.md`
in the repo. The goal of this ticket is to automate the full pipeline end to end —
from content input to Instagram-ready PNGs — and build the foundation for future
features including animations and multiple aspect ratios.

The developer has already built a Carousel Studio app on Netlify that demonstrates
themes and ratio switching. That app is the right direction — this ticket expands
it into a complete production pipeline.

---

## Current State

The existing system works like this:

```
1. Content is written manually or generated by Claude (AI)
2. Claude outputs an HTML file using shared styles.css
3. HTML file is placed in the repo folder
4. screenshot.js (Playwright) is run locally
5. Playwright opens each HTML file in headless Chrome
6. It calls window.goToSlide(n) per slide and screenshots each one
7. PNGs are saved to output/ folder locally
8. PNGs are manually uploaded to Buffer and scheduled
```

All HTML files share a single `styles.css` with design tokens:
- Backgrounds: `--bg #111009`, `--bg2 #181612`, `--bg3 #1d1b12`
- Gold accent: `--gold #C9A84C`, `--gdim #7a6530`
- White text: `--white #EDE8DC`
- Body text: `--gray #68665e`
- Fonts: DM Serif Display (headlines), DM Sans (body)
- Canvas: 1080×1080px (square), with portrait 1080×1350 coming

Every slide exposes `window.goToSlide(n)` for the screenshot script.
Every carousel has exactly 10 slides.

## What Needs to Be Built

### Phase 1 — Hosted Screenshot Renderer (replace local Playwright)

**Goal:** Move screenshotting off local machines entirely.

Build a small serverless or always-on render service that:

1. Accepts a POST request with either:
   - A carousel filename (e.g. `carousel-v5-compound-loop.html`)
   - Or a full HTML string
2. Opens the carousel in headless Chromium (Playwright or Puppeteer)
3. Iterates through all slides using `window.goToSlide(n)`
4. Screenshots each slide at the correct dimensions
5. Returns a ZIP file of PNGs, or uploads them to a storage bucket and returns URLs

**Suggested stack:** Node.js + Playwright on Railway, Render, or a serverless function
(Vercel Edge Functions won't work — Playwright needs a full Node environment).

**Endpoint example:**
```
POST /render
{
  "source": "https://majajecar.github.io/encoded-carousels/carousel-v5-compound-loop.html",
  "slides": 10,
  "width": 1080,
  "height": 1080
}

Response:
{
  "images": [
    "https://storage.../v5-compound-loop/slide-01.png",
    ...
  ]
}
```

**Important:** The renderer must wait for `networkidle` and an additional 1800ms after
page load for Google Fonts to render correctly before taking screenshots.
Per-slide delay after `goToSlide()` should be at least 420ms to allow CSS transitions.

### Phase 2 — Content Input API

**Goal:** Accept carousel content and produce an HTML file automatically.

Claude (via the Anthropic API) currently writes carousel HTML directly. The pipeline
should expose an endpoint that accepts structured content and returns a rendered HTML file.

**Option A — Structured JSON input (simpler):**
```
POST /generate
{
  "series": "The Compound Loop",
  "chapter": "The Frequency Era",
  "slides": [
    {
      "type": "cover",
      "eyebrow": "The Compound Loop",
      "headline": "Every confirming\nexperience compounds.",
      "subhead": "Why belief-based programs intensify."
    },
    {
      "type": "comparison",
      "eyebrow": "The Trap",
      "headline": "This is why more\ninformation doesn't work.",
      "left": { "label": "New Information", "title": "Filtered out", "body": "..." },
      "right": { "label": "The Loop", "title": "Confirms and encodes", "body": "..." }
    }
    // ... up to 10 slides
  ],
  "cta_word": "LOOP"
}
```

**Option B — Claude-generated HTML (more flexible):**
Claude receives a system prompt containing the full design system (all CSS classes,
layout rules, density limits) and outputs a complete HTML file directly.
The endpoint forwards content to Claude API and returns the HTML.

Both options are valid. Option B is more flexible but requires prompt engineering.
Option A is more predictable and easier to validate. Recommend building Option A first
and adding Option B as a layer on top.

**Slide types to support (with their HTML structures — see styles.css for all classes):**
- `cover` — eyebrow + `.hxl` headline + `.sub` subhead + `.brand`
- `statement` — eyebrow + `.hl` headline only
- `statement_sub` — `.hl` headline + `.body` + optional `.vbar` pull quote
- `list` — `.hm` + `.rule` + up to 4 `.nrow` items + optional footer
- `comparison` — `.hm` + `.g2` grid (`.card` + `.card.em`)
- `sequence` — `.hm` + `.seq` column with 3 cards + down arrows (SVG)
- `hflow` — `.hm` + `.hflow` with 3 `.hcard` items + right arrows (SVG)
- `quote` — `.qm` + `.qt` + `.qa` + `.spacer` both sides
- `stat` — eyebrow + `.sn` number + `.su` unit + caption + `.g3h` grid
- `closing` — `.brand` + eyebrow + `.hl` + `.rule` + CTA comment line

### Phase 3 — GitHub Integration

**Goal:** Every new carousel HTML pushed to the repo triggers automatic screenshotting.

Set up a GitHub Actions workflow that:

1. Triggers on push to `main` when any `carousel-*.html` file changes
2. Identifies which carousel files were added or modified
3. Calls the Phase 1 render service for each changed file
4. Saves the resulting PNGs to a designated storage location
   (Google Drive folder, S3 bucket, or GitHub Release assets)
5. Posts a Slack or email notification with links to the new PNGs

**GitHub Actions workflow sketch:**
```yaml
name: Render Carousels

on:
  push:
    branches: [main]
    paths:
      - 'carousel-*.html'

jobs:
  render:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with:
          fetch-depth: 2

      - name: Get changed carousel files
        id: changed
        run: |
          git diff --name-only HEAD~1 HEAD | grep 'carousel-.*\.html' > changed.txt
          cat changed.txt

      - name: Render each carousel
        run: |
          while read file; do
            curl -X POST $RENDER_SERVICE_URL/render \
              -H "Content-Type: application/json" \
              -d "{\"source\": \"https://majajecar.github.io/encoded-carousels/$file\", \"slides\": 10}"
          done < changed.txt
```

### Phase 4 — Scheduling Integration (Buffer)

**Goal:** Rendered PNGs automatically land in a scheduling queue.

Buffer has an API. Once PNGs are rendered and stored:

1. Call Buffer API to create a draft post with the 10 slides in order
2. Set it as a draft (not auto-published — a human reviews before it goes live)
3. Include a suggested caption (generated by Claude or templated)

This keeps a human in the approval loop while removing the manual upload step.

**Buffer API reference:** https://buffer.com/developers/api

### Phase 5 — Aspect Ratio Support

**Current problem:** The Netlify app already has ratio switching, but it currently
scales/crops the same layout rather than rearranging elements. This needs to be fixed.

**Correct approach:** Each ratio is a different layout, not a scaled version.

**Ratios to support:**
- `1:1` — 1080×1080 — feed carousels (current)
- `4:5` — 1080×1350 — feed single posts, highest reach format
- `9:16` — 1080×1920 — Stories, Reels cover

**How to implement correctly:**

The design system already uses CSS custom properties. Add canvas size variables:

```css
:root {
  --canvas-w: 1080px;
  --canvas-h: 1080px;
  --slide-pad-v: 88px;
  --slide-pad-h: 86px;
}
```

For portrait (4:5), the layout adjustments needed per slide type:
- Cover: photo can be larger, headline can go bigger
- Statement: more vertical breathing room — increase spacer, bump headline size
- Comparison (2-column): stays 2-column but cards are taller, more body text fits
- Sequence (3 stacked cards): works naturally in portrait — more room per card
- Quote: more vertical centering space — looks better in portrait actually
- Stat: large number gets more space, caption can be longer
- Closing: more room between brand and headline

For 9:16 Stories: elements need to be centered more aggressively,
font sizes increase, card layouts may need to stack where they currently sit side by side.

**Recommended approach:**
Build a responsive override system using a `data-ratio` attribute on `<body>`:
```css
body[data-ratio="4:5"] {
  --canvas-h: 1350px;
  /* layout overrides for portrait */
}

body[data-ratio="9:16"] {
  --canvas-h: 1920px;
  /* layout overrides for stories */
}
```

The render service passes `width` and `height` to the renderer, which sets
`data-ratio` on `<body>` before screenshotting.

### Phase 6 — Animation Support (Future, Architecture Only)

**Do not build now. Design the architecture to support it.**

Future animated carousels will use CSS animations or a JS animation library
(Framer Motion, GSAP, or CSS keyframes). The render service will need to:

1. Detect whether a carousel has `data-animated="true"` on the body
2. For animated carousels: instead of screenshotting, use a video capture approach
   (record each slide for its animation duration, export as MP4 or GIF)
3. Instagram supports video carousels — each slide becomes a short video clip

**Key architectural decision to make now:**
- Keep the render service stateless (accepts a URL, returns images/video)
- Add `animated: true` and `duration_per_slide: 3000` fields to the render API
  so the service knows to use video capture instead of screenshots
- Store animation keyframes in a separate `animations.css` file that is
  conditionally loaded (not part of `styles.css`)

This way, static and animated carousels share the same HTML structure
and the same render service — only the capture method changes.

## Design System Reference

All carousel HTML files follow these rules:

1. Link to `styles.css` (never embed CSS inline)
2. Link to Google Fonts: DM Sans + DM Serif Display
3. `html, body` set to exact canvas dimensions, `overflow:hidden`
4. `.slide` at `position:absolute`, exact canvas dimensions, `padding:88px 86px`
5. First slide has `class="slide active"`, all others have `class="slide"`
6. All slides expose:
```js
const allSlides = document.querySelectorAll('.slide');
window.totalSlides = allSlides.length;
window.goToSlide = function(n) {
  allSlides.forEach(s => s.classList.remove('active'));
  allSlides[n].classList.add('active');
};
```

**CSS transition:** Slides fade in/out with `opacity 0.3s` — screenshot delay
after `goToSlide()` must be at least 420ms to capture the fully visible state.

**Font load timing:** Google Fonts must be fully loaded before first screenshot.
Use `networkidle` wait + 1800ms additional delay after page load.

## Branding Rules (for any UI the developer builds)

- Brand name: **Encoded.Ai** (capital E, capital A — always)
- Brand mark appears ONCE per slide, on cover and closing slides only
- Gold used sparingly — one emphasized element per slide maximum
- No Save/Share/Comment action blocks on slides
- Closing slides end with: `Comment [WORD] — we'll send the deep-dive`
